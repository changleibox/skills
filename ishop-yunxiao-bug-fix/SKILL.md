# 云效DevOps工作项处理指南

## 核心原则

| 原则 | 说明 |
|------|------|
| 确认优先 | 不确定的事一定要问用户，不擅自做决定。所有涉及外部操作（修改代码、提交评论、更新状态、转交负责人）的步骤，必须先向用户展示具体内容并获得确认后才能执行 |
| 责任范围限定 | 仅处理分配给当前用户的Bug工作项，不主动介入他人工作项 |
| 评论格式约束 | 云效平台不支持Markdown渲染，评论内容必须使用纯文本格式 |
| 必填参数要求 | 查询工作项时必须提供 `spaceId`（项目ID）和 `category`（工作项类型）参数 |

## 操作流程

### 步骤一：获取基础信息

```text
mcp_RAM_get_current_organization_info  --> 获取组织ID(organizationId)和用户ID(userId)
mcp_RAM_search_projects                 --> 获取项目列表及对应的spaceId
```

**默认项目规则**：
- 如果用户**未指定具体项目名称**，默认使用 **"iShop ERP 项目"** 作为目标项目
- 调用 `mcp_RAM_search_projects` 时使用关键词 "iShop ERP" 搜索，获取对应的 `spaceId`
- 如果用户**明确指定了项目名称**，则按用户指定的项目查询
- 如果默认项目未找到，提示用户并询问要查看哪个项目

### 步骤二：查询工作项列表

查询当前用户分配的Bug工作项：

```text
mcp_RAM_search_workitems 参数说明：
  - organizationId: 组织ID（必填）
  - spaceId: 项目空间ID（必填）
  - category: "Bug"（必填，工作项类型）
  - assignedTo: "self" 或具体用户ID（指定负责人）
  - status: 状态ID列表，多个用逗号分隔（可选）
```

**默认状态过滤规则**：
- 默认只查询**需要处理的Bug**，即状态为：待确认(28)、待处理(100005)、重新打开(30)、处理中(100010)
- 查询时 `status` 参数默认传 `"28,100005,30,100010"`
- 已修复(29)和已关闭(100085)的Bug默认**不查询**，因为这些是已经处理过的
- 如果用户**明确要求**查看特定状态的Bug（如"看看已修复的bug"、"查看所有状态的bug"），则按用户指定的状态查询

**工作项状态ID对照表**：

| 状态名称 | 状态ID | 说明 | 默认查询 |
|----------|--------|------|----------|
| 待确认 | 28 | 新建Bug的初始状态 | ✅ |
| 待处理 | 100005 | 已确认待开发处理 | ✅ |
| 重新打开 | 30 | 验证未通过被重新打开 | ✅ |
| 处理中 | 100010 | 开发人员正在修复 | ✅ |
| 已修复 | 29 | 修复完成待验证 | ❌ |
| 已关闭 | 100085 | 验证通过或无需修复 | ❌ |

### 步骤二A：高级搜索（可选）

当用户触发高级搜索（如"搜索bug"、"查找bug"、"筛选bug"）或在步骤二后主动要求进一步过滤时，进入高级搜索流程：

1. **选择过滤维度**：使用 `AskUserQuestion` 工具提供过滤维度选项（支持多选）：
   - "关键词搜索"：按标题或描述中的关键词过滤
   - "优先级过滤"：按高/中/低优先级过滤
   - "时间范围"：按创建时间范围过滤
   - "状态过滤"：按指定状态过滤
   - "创建人过滤"：按Bug创建人过滤

2. **逐项收集参数**：根据用户选择的维度，依次收集具体参数值：
   - 关键词搜索 → 使用 `AskUserQuestion` 询问搜索关键词（用户通过“Other”选项输入）
   - 优先级过滤 → 使用 `AskUserQuestion` 提供选项：高、中、低（支持多选）
   - 时间范围 → 使用 `AskUserQuestion` 询问起止时间（用户通过“Other”选项输入，格式 YYYY-MM-DD）
   - 状态过滤 → 使用 `AskUserQuestion` 提供全部6种状态选项（支持多选）
   - 创建人过滤 → 使用 `AskUserQuestion` 询问创建人姓名（用户通过“Other”选项输入）

3. **参数映射与查询**：将收集到的参数映射到 `mcp_RAM_search_workitems` 接口：

| 过滤维度 | 映射参数 | 说明 |
|----------|----------|------|
| 关键词搜索 | `subjectDescription` | 在标题和描述中搜索 |
| 优先级过滤 | `priority` | 优先级ID，多个用逗号分隔 |
| 时间范围 | `createdAfter` / `createdBefore` | 创建时间范围 |
| 状态过滤 | `status` | 状态ID，多个用逗号分隔 |
| 创建人过滤 | `creator` | 创建人用户ID（需先通过 `mcp_RAM_search_organization_members` 查询） |

4. 查询结果展示后，进入步骤三的循环处理流程

### 步骤三：循环处理Bug列表（核心循环入口）

此步骤是Bug处理循环的入口，维护一个"待处理Bug列表"，每完成一个Bug后自动回到此步骤继续处理下一个，直到列表清空或用户主动退出。

**处理记录追踪**：维护一个"处理记录表"，每完成一个Bug时记录其标题和处理方式（修复/转交/无法复现/不是问题/需要更多信息/重复Bug/延期处理），用于循环结束时生成汇总报告。

**循环逻辑**：

1. **检查待处理列表**：
   - 列表为空 → 生成处理汇总报告后流程结束，报告格式：
     ```
     ===== Bug处理汇总 =====
     处理总数：N 个
     
     处理方式分布：
     - 修复：X 个
     - 转交：X 个
     - 无法复现：X 个
     - 不是问题：X 个
     - 需要更多信息：X 个
     - 重复Bug：X 个
     - 延期处理：X 个
     
     处理明细：
     1. [Bug标题] → 修复
     2. [Bug标题] → 转交给XXX
     3. [Bug标题] → 无法复现
     ...
     =======================
     ```
     注意：如果没有处理任何Bug（用户在第一轮就选择"暂不处理"），则仅告知"没有处理任何Bug"，不生成报告。
   - 列表不为空 → 继续下方流程

2. **展示剩余Bug**：以简洁列表形式展示所有待处理Bug，每条包含：编号、标题、优先级、当前状态。如果是循环回来的（非首次），额外提示"已完成 N 个，剩余 M 个"

3. **询问用户选择**：使用 `AskUserQuestion` 工具，选项包括：
   - 各个Bug的标题作为选项（支持选择具体要修复的Bug）
   - "暂不处理"选项

4. **分支处理**：
   - 用户选择"暂不处理" → 流程结束
   - 用户选择了某个Bug → 将该Bug从待处理列表中移除，进入步骤四

### 步骤四：获取工作项详情并判断处理方式

```text
mcp_RAM_get_work_item 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项唯一标识（必填）
```

**获取附件（如果有的话）**：
```text
mcp_RAM_list_workitem_attachments 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
```

**获取历史评论**：
```text
mcp_RAM_list_work_item_comments 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
```

获取详情后，向用户展示Bug的完整信息：

1. **基本信息**：标题、描述、复现步骤、优先级、创建人等
2. **历史评论**：如果有历史评论，按时间顺序展示（评论人、时间、内容），可能包含重要的上下文信息
3. **附件展示**：
   - 如果有截图/图片附件，使用 `mcp_RAM_get_workitem_file` 获取下载链接
   - 展示图片给用户查看（使用 Markdown 图片语法：`![描述](图片URL)`）
   - 列出其他附件文件名和大小

**附件处理流程**：
1. 调用 `mcp_RAM_list_workitem_attachments` 获取附件列表
2. 遍历附件列表，识别图片类型（.png, .jpg, .jpeg, .gif, .webp 等）
3. 对于图片附件：
   - 调用 `mcp_RAM_get_workitem_file` 获取临时下载URL
   - 使用 Markdown 语法展示图片：`![Bug截图](下载URL)`
4. 对于其他附件：列出文件名、大小、上传时间

**相似Bug检测**（自动执行，无需用户触发）：

1. 从当前Bug的标题中提取核心关键词（去除常见词如"页面"、"功能"、"问题"等）
2. 调用 `mcp_RAM_search_workitems` 使用 `subjectDescription` 参数搜索可能相似的Bug：
   ```text
   mcp_RAM_search_workitems 参数说明：
     - organizationId: 组织ID（必填）
     - spaceId: 项目空间ID（必填）
     - category: "Bug"（必填）
     - subjectDescription: 提取的核心关键词
   ```
3. 从搜索结果中排除当前Bug自身，筛选标题或描述与当前Bug高度相关的条目
4. 如果找到相似Bug（最多展示前3个），以提示信息形式展示：
   ```
   💡 发现可能相关的Bug：
   1. [BUG-XXX] <标题> | 状态：<状态> | 创建时间：<时间>
   2. [BUG-XXX] <标题> | 状态：<状态> | 创建时间：<时间>
   以上Bug可能与当前Bug相关，供处理时参考。
   ```
5. 如果没有找到相似Bug，静默跳过，不展示任何提示

展示完详情后，使用 `AskUserQuestion` 工具询问处理方式，选项包括：

- **"修复此Bug"**：进入步骤四A（执行修复）
- **"不是我负责的，转给其他人"**：进入步骤四B（转交负责人）
- **"无法复现"**：进入步骤四C（退回创建人）
- **"不是问题/设计如此"**：进入步骤四D（标记为非问题）
- **"需要更多信息"**：进入步骤四E（退回创建人补充信息）
- **"重复Bug"**：进入步骤四F（标记为重复并关联原始Bug）
- **"延期处理"**：进入步骤四G（推迟到后续版本修复）

#### 步骤四A：执行修复

1. **更新状态为"处理中"**：如果当前Bug状态不是"处理中(100010)"，先将状态更新为"处理中"，表示开始修复：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - status: "100010"（处理中）
   ```
   如果状态更新失败（工作流限制），提示用户但不影响后续修复流程
2. 分析Bug描述、复现步骤等信息
3. 在代码库中定位问题代码，形成修复方案
4. **确认修复方案**：向用户说明问题根因、修复思路、涉及的文件和改动点，使用 `AskUserQuestion` 工具确认是否按此方案执行，选项包括：
   - "确认，按此方案修复"
   - "我有其他想法"（用户通过"Other"选项补充修改意见）
5. 用户确认后，实施代码修复
6. **修复确认循环**：代码修改完成后，使用 `AskUserQuestion` 工具向用户确认"代码修改是否已完成？"，选项包括：
   - "已完成，进入下一步"：退出循环，进入步骤五
   - "还没改完，继续修改"：用户通过 "Other" 选项描述还需要修改的内容，继续修改后再次确认，**重复此步骤直到用户确认完成**
7. 用户确认修复完成后，进入步骤五

#### 步骤四B：转交负责人（不是我负责的Bug）

当Bug内容不属于当前用户的职责范围时：

1. **询问是否需要添加评论**：使用 `AskUserQuestion` 工具询问"转交前是否需要添加评论说明原因？"，选项包括：
   - "需要添加评论"：进入步骤2的评论流程，评论完成后再继续转交流程
   - "自动生成评论并转交"：跳到步骤3，自动根据测试情况生成评论
   - "不添加评论，直接转交"：跳到步骤8继续（不推荐）

2. **评论流程（如果用户选择需要评论）**：
   - 使用 `AskUserQuestion` 工具询问用户评论内容（用户通过"Other"选项输入）
   - 展示生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 用户确认后的纯文本评论内容（必填）
     ```
   - 跳到步骤8继续转交流程

3. **自动生成评论（如果用户选择自动生成）**：
   - 根据Bug描述内容，自动生成转交说明评论
   - 评论内容应包含：
     - Bug涉及的功能模块/页面
     - 为什么不属于自己负责（如"此问题属于XX模块")
   - 展示自动生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交并转交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
     - "取消"（取消则进入步骤八，跳过此Bug）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 自动生成的转交说明评论内容（必填）
     ```

4. **智能推荐负责人**（自动执行）：
   - 分析当前Bug标题和描述中的模块/功能关键词（如"销售开单"、"商品列表"、"库存管理"等）
   - 调用 `mcp_RAM_search_workitems` 查询最近已修复（status=29）的包含相同关键词的Bug，提取其负责人信息
   - 如果找到推荐人选：使用 `AskUserQuestion` 展示推荐人选作为选项，同时提供"其他人"选项（用户通过“Other”输入自定义姓名）
   - 如果无法推荐（关键词不明确或无历史数据）：使用 `AskUserQuestion` 直接询问"请输入要转交的负责人姓名"（用户通过“Other”选项输入）

5. 根据用户提供的姓名，调用组织成员查询接口查找对应用户ID：
   ```text
   mcp_RAM_search_organization_members 参数说明：
     - organizationId: 组织ID（必填）
     - query: 用户提供的负责人姓名
   ```

6. 如果查到多个匹配结果，再次使用 `AskUserQuestion` 让用户从匹配列表中选择确切的人

7. **确认转交操作**：向用户展示即将执行的操作摘要（"将Bug [标题] 的负责人转交给 [新负责人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认转交"
   - "取消"（取消则进入步骤八，跳过此Bug）

8. 用户确认后，更新工作项负责人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 新负责人用户ID
   ```

9. 告知用户已成功转交，进入步骤八（继续处理下一个Bug）

**转交评论模板**：

```text
转交原因：
此问题属于[模块名称/功能名称]，由[负责人姓名/团队]负责处理。

问题描述：
<Bug标题或简要描述>

建议转交给：
[建议的负责人姓名]
```

#### 步骤四C：无法复现（退回创建人）

当Bug无法复现时，将Bug退回给创建人，并附上详细的复现尝试说明：

1. 从步骤四获取的工作项详情中提取**创建人ID**（creator字段）
2. 使用 `AskUserQuestion` 工具询问用户复现情况，提示用户描述：
   - 做过哪些测试操作
   - 测试的环境和条件
   - 为什么无法复现
   （用户通过"Other"选项输入自定义文本）
3. 根据用户描述，按照"无法复现说明模板"生成评论草稿
4. **确认评论内容和退回操作**：向用户展示生成的评论全文，并说明即将执行的操作（"提交以上评论，并将负责人转交给创建人 [创建人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并退回"
   - "修改内容"（用户通过"Other"选项提供修改意见，修改后再次确认）
   - "取消"（取消则进入步骤八，跳过此Bug）
5. 用户确认后，提交无法复现说明评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 按照下方"无法复现说明模板"填写的纯文本内容（必填）
   ```
6. 将负责人改为创建人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 创建人用户ID
   ```
7. 告知用户已退回给创建人，进入步骤八（继续处理下一个Bug）

**无法复现说明模板（必须100%遵守）**：

```text
复现结果：
无法复现

测试环境：
<描述测试使用的设备、系统版本、应用版本等环境信息>

测试操作：
<逐条列出按照Bug描述执行的测试步骤，格式为"N. 操作描述 -> 实际结果"，每步占一行>

结论：
<总结无法复现的原因判断，如"按照描述步骤多次测试均未出现该问题，建议提供更详细的复现条件"等>
```

**无法复现说明示例**：

```text
复现结果：
无法复现

测试环境：
```

## 错误处理与重试机制

所有涉及云效API调用的步骤（评论提交、状态更新、负责人转交、附件上传等）均适用以下错误处理策略：

| 错误类型 | 典型表现 | 处理策略 |
|----------|----------|----------|
| 网络/临时错误 | 请求超时、服务端5xx错误 | 自动重试1次（间隔3秒），仍失败则提示用户稍后重试 |
| 权限/工作流错误 | 403禁止、状态流转被拒绝 | 不重试，展示具体限制原因，引导用户到云效Web控制台手动操作 |
| 参数错误 | 400请求无效、404资源不存在 | 不重试，展示错误详情，引导用户检查输入参数 |

**错误处理流程**：

1. API调用返回错误后，首先判断错误类型
2. 如果是网络/临时错误：
   - 提示用户"操作暂时失败，正在自动重试..."
   - 等待3秒后重试1次
   - 重试成功则继续后续流程
   - 重试仍失败则提示用户"操作失败，建议稍后重试或在云效Web端手动操作"
3. 如果是权限/工作流错误：
   - 提示用户具体的限制原因（如"当前状态不允许直接流转到已修复，请在云效Web端操作"）
   - 跳过当前操作，继续后续流程（不阻塞整体处理）
4. 如果是参数错误：
   - 展示错误详情
   - 询问用户是否要重新输入参数或跳过当前操作

**重要原则**：错误不应阻断整个Bug处理流程。即使某个API调用失败，也应允许用户继续处理其他步骤或其他Bug。

## 常见问题解答

**Q1：如何查询"我参与评论过的"工作项？**

A：云效API暂不支持此筛选条件，请登录云效Web控制台，在「工作项 → 我的」页面查看历史记录。

**Q2：工作项状态更新失败如何处理？**

A：状态流转受工作流严格限制，API可能无权限直接修改。建议先通过评论记录修复情况，然后在Web控制台手动更新状态。

**Q3：查询工作项时提示"项目不存在"怎么办？**

A：请确认 `spaceId` 参数正确。应先调用 `mcp_RAM_search_projects` 获取当前用户可访问的项目列表。
