# 云效DevOps工作项处理指南

## 核心原则

| 原则 | 说明 |
|------|------|
| 确认优先 | 不确定的事一定要问用户，不擅自做决定。所有涉及外部操作（修改代码、提交评论、更新状态、转交负责人）的步骤，必须先向用户展示具体内容并获得确认后才能执行 |
| 责任范围限定 | 仅处理分配给当前用户的Bug工作项，不主动介入他人工作项 |
| 评论格式约束 | 云效平台不支持Markdown渲染，评论内容必须使用纯文本格式 |
| 必填参数要求 | 查询工作项时必须提供 `spaceId`（项目ID）和 `category`（工作项类型）参数 |

## 操作流程

### 步骤一：获取基础信息

```text
mcp_RAM_get_current_organization_info  --> 获取组织ID(organizationId)和用户ID(userId)
mcp_RAM_search_projects                 --> 获取项目列表及对应的spaceId
```

**默认项目规则**：
- 如果用户**未指定具体项目名称**，默认使用 **"iShop ERP 项目"** 作为目标项目
- 调用 `mcp_RAM_search_projects` 时使用关键词 "iShop ERP" 搜索，获取对应的 `spaceId`
- 如果用户**明确指定了项目名称**，则按用户指定的项目查询
- 如果默认项目未找到，提示用户并询问要查看哪个项目

### 步骤二：查询工作项列表

查询当前用户分配的Bug工作项：

```text
mcp_RAM_search_workitems 参数说明：
  - organizationId: 组织ID（必填）
  - spaceId: 项目空间ID（必填）
  - category: "Bug"（必填，工作项类型）
  - assignedTo: "self" 或具体用户ID（指定负责人）
  - status: 状态ID列表，多个用逗号分隔（可选）
```

**默认状态过滤规则**：
- 默认只查询**需要处理的Bug**，即状态为：待确认(28)、待处理(100005)、重新打开(30)、处理中(100010)
- 查询时 `status` 参数默认传 `"28,100005,30,100010"`
- 已修复(29)和已关闭(100085)的Bug默认**不查询**，因为这些是已经处理过的
- 如果用户**明确要求**查看特定状态的Bug（如"看看已修复的bug"、"查看所有状态的bug"），则按用户指定的状态查询

**工作项状态ID对照表**：

| 状态名称 | 状态ID | 说明 | 默认查询 |
|----------|--------|------|----------|
| 待确认 | 28 | 新建Bug的初始状态 | ✅ |
| 待处理 | 100005 | 已确认待开发处理 | ✅ |
| 重新打开 | 30 | 验证未通过被重新打开 | ✅ |
| 处理中 | 100010 | 开发人员正在修复 | ✅ |
| 已修复 | 29 | 修复完成待验证 | ❌ |
| 已关闭 | 100085 | 验证通过或无需修复 | ❌ |

### 步骤二A：高级搜索（可选）

当用户触发高级搜索（如"搜索bug"、"查找bug"、"筛选bug"）或在步骤二后主动要求进一步过滤时，进入高级搜索流程：

1. **选择过滤维度**：使用 `AskUserQuestion` 工具提供过滤维度选项（支持多选）：
   - "关键词搜索"：按标题或描述中的关键词过滤
   - "优先级过滤"：按高/中/低优先级过滤
   - "时间范围"：按创建时间范围过滤
   - "状态过滤"：按指定状态过滤
   - "创建人过滤"：按Bug创建人过滤

2. **逐项收集参数**：根据用户选择的维度，依次收集具体参数值：
   - 关键词搜索 → 使用 `AskUserQuestion` 询问搜索关键词（用户通过“Other”选项输入）
   - 优先级过滤 → 使用 `AskUserQuestion` 提供选项：高、中、低（支持多选）
   - 时间范围 → 使用 `AskUserQuestion` 询问起止时间（用户通过“Other”选项输入，格式 YYYY-MM-DD）
   - 状态过滤 → 使用 `AskUserQuestion` 提供全部6种状态选项（支持多选）
   - 创建人过滤 → 使用 `AskUserQuestion` 询问创建人姓名（用户通过“Other”选项输入）

3. **参数映射与查询**：将收集到的参数映射到 `mcp_RAM_search_workitems` 接口：

| 过滤维度 | 映射参数 | 说明 |
|----------|----------|------|
| 关键词搜索 | `subjectDescription` | 在标题和描述中搜索 |
| 优先级过滤 | `priority` | 优先级ID，多个用逗号分隔 |
| 时间范围 | `createdAfter` / `createdBefore` | 创建时间范围 |
| 状态过滤 | `status` | 状态ID，多个用逗号分隔 |
| 创建人过滤 | `creator` | 创建人用户ID（需先通过 `mcp_RAM_search_organization_members` 查询） |

4. 查询结果展示后，进入步骤三的循环处理流程

### 步骤三：循环处理Bug列表（核心循环入口）

此步骤是Bug处理循环的入口，维护一个"待处理Bug列表"，每完成一个或多个Bug后自动回到此步骤继续处理，直到列表清空或用户主动退出。

**处理记录追踪**：维护一个"处理记录表"，每完成一个或一批Bug时记录其标题和处理方式（修复/转交/无法复现/不是问题/需要更多信息/重复Bug/延期处理），用于循环结束时生成汇总报告。处理记录表支持一次性记录多个Bug的处理结果（批量/流水线/分组模式下）。

**循环逻辑**：

1. **检查待处理列表**：
   - 列表为空 → 生成处理汇总报告后流程结束，报告格式：
     ```
     ===== Bug处理汇总 =====
     处理总数：N 个
     
     处理方式分布：
     - 修复：X 个
     - 转交：X 个
     - 无法复现：X 个
     - 不是问题：X 个
     - 需要更多信息：X 个
     - 重复Bug：X 个
     - 延期处理：X 个
     
     处理明细：
     1. [Bug标题] → 修复
     2. [Bug标题] → 转交给XXX
     3. [Bug标题] → 无法复现
     ...
     =======================
     ```
     注意：如果没有处理任何Bug（用户在第一轮就选择"暂不处理"），则仅告知"没有处理任何Bug"，不生成报告。
   - 列表不为空 → 继续下方流程

2. **展示剩余Bug**：以简洁列表形式展示所有待处理Bug，每条包含：编号、标题、优先级、当前状态。如果是循环回来的（非首次），额外提示"已完成 N 个，剩余 M 个"

3. **询问用户选择**：使用 `AskUserQuestion`（multiSelect: true）工具，选项包括：
   - 各个Bug的标题作为选项（支持选择一个或多个Bug）
   - [全选]（选中当前待处理列表中的所有Bug）
   - [暂不处理]

4. **分支处理**：
   - 用户选择"暂不处理" → 流程结束
   - 用户选择了**1个Bug**（单选） → 将该Bug从待处理列表中移除，进入步骤四（沿用原有步骤四~八流程，无任何变化）
   - 用户选择了**多个Bug**（多选或全选） → 进入**步骤三A：处理模式选择**

**列表更新规则**：
- 单Bug模式（步骤四~八）：处理完毕后，从待处理列表中移除1个Bug
- 多Bug模式（批量/流水线/分组）：处理完毕后，将所有已处理的Bug从待处理列表中批量移除

### 步骤三A：处理模式选择（多Bug场景）

> 仅当用户在步骤三中选择了**多个Bug**时触发此步骤

使用 `AskUserQuestion` 询问处理模式：

选项：
- [批量操作] - 对所有选中Bug执行相同操作（适合同类Bug统一处理）
- [逐个精简处理] - 按顺序快速处理每个Bug（适合不同类型Bug混合处理）
- [智能分组处理] - AI自动按模块分组后分组处理（适合大量Bug快速归类）

分支处理：
- 用户选择"批量操作" → 进入步骤三B
- 用户选择"逐个精简处理" → 进入步骤三C
- 用户选择"智能分组处理" → 进入步骤三D

> **[执行前必读]** 用户选择处理模式后，必须使用 `read_file` 工具读取同目录下的 `SKILL-multi-bug.md` 文件，获取步骤三B（批量操作）、三C（流水线模式）、三D（智能分组处理）的完整操作流程。文件路径为技能安装目录下的 `SKILL-multi-bug.md`。

### 步骤四：获取工作项详情并判断处理方式

```text
mcp_RAM_get_work_item 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项唯一标识（必填）
```

**获取附件（如果有的话）**：
```text
mcp_RAM_list_workitem_attachments 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
```

**获取历史评论**：
```text
mcp_RAM_list_work_item_comments 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
```

获取详情后，向用户展示Bug的完整信息：

1. **基本信息**：标题、描述、复现步骤、优先级、创建人等
2. **历史评论**：如果有历史评论，按时间顺序展示（评论人、时间、内容），可能包含重要的上下文信息
3. **附件展示**：
   - 如果有截图/图片附件，使用 `mcp_RAM_get_workitem_file` 获取下载链接
   - 展示图片给用户查看（使用 Markdown 图片语法：`![描述](图片URL)`）
   - 列出其他附件文件名和大小

**附件处理流程**：
1. 调用 `mcp_RAM_list_workitem_attachments` 获取附件列表
2. 遍历附件列表，识别图片类型（.png, .jpg, .jpeg, .gif, .webp 等）
3. 对于图片附件：
   - 调用 `mcp_RAM_get_workitem_file` 获取临时下载URL
   - 使用 Markdown 语法展示图片：`![Bug截图](下载URL)`
4. 对于其他附件：列出文件名、大小、上传时间

**相似Bug检测**（自动执行，无需用户触发）：

1. 从当前Bug的标题中提取核心关键词（去除常见词如"页面"、"功能"、"问题"等）
2. 调用 `mcp_RAM_search_workitems` 使用 `subjectDescription` 参数搜索可能相似的Bug：
   ```text
   mcp_RAM_search_workitems 参数说明：
     - organizationId: 组织ID（必填）
     - spaceId: 项目空间ID（必填）
     - category: "Bug"（必填）
     - subjectDescription: 提取的核心关键词
   ```
3. 从搜索结果中排除当前Bug自身，筛选标题或描述与当前Bug高度相关的条目
4. 如果找到相似Bug（最多展示前3个），以提示信息形式展示：
   ```
   💡 发现可能相关的Bug：
   1. [BUG-XXX] <标题> | 状态：<状态> | 创建时间：<时间>
   2. [BUG-XXX] <标题> | 状态：<状态> | 创建时间：<时间>
   以上Bug可能与当前Bug相关，供处理时参考。
   ```
5. 如果没有找到相似Bug，静默跳过，不展示任何提示

展示完详情后，使用 `AskUserQuestion` 工具询问处理方式，选项包括：

- **"修复此Bug"**：进入步骤四A（执行修复）
- **"不是我负责的，转给其他人"**：进入步骤四B（转交负责人）
- **"无法复现"**：进入步骤四C（退回创建人）
- **"不是问题/设计如此"**：进入步骤四D（标记为非问题）
- **"需要更多信息"**：进入步骤四E（退回创建人补充信息）
- **"重复Bug"**：进入步骤四F（标记为重复并关联原始Bug）
- **"延期处理"**：进入步骤四G（推迟到后续版本修复）

> **[执行前必读]** 用户选择处理方式后，必须使用 `read_file` 工具读取同目录下的 `SKILL-handling.md` 文件，获取对应步骤（四A~四G）的详细操作流程和评论模板。文件路径为技能安装目录下的 `SKILL-handling.md`。
> - 步骤四A（修复）完成后，回到本文件的步骤五继续。
> - 步骤四B~四G完成后，直接回到本文件的步骤八。

### 步骤五：修复完成后询问是否更新云效状态

代码修复完成后，使用 `AskUserQuestion` 工具询问用户后续操作，选项包括：
- "提交评论并更新状态为已修复"：执行步骤六（提交修复说明）+ 步骤七（更新状态为已修复，状态ID: 29），然后进入步骤八
- "仅提交评论"：仅执行步骤六（提交修复说明），不修改状态，然后进入步骤八
- "上传附件"：进入步骤六A（上传附件），上传完成后回到本步骤继续选择
- "跳过"：不提交评论也不更新状态，直接进入步骤八（继续处理下一个Bug）

### 步骤六：提交修复说明（工作项评论）

**格式约束**：
- 评论内容必须为纯文本格式，不支持Markdown语法渲染
- 必须严格使用下方模板的四个固定标题，标题文字不可修改、不可省略、不可调换顺序
- 每个标题后紧跟冒号，冒号后换行书写具体内容
- 四个部分之间用一个空行分隔
- 禁止添加模板以外的标题或分节

**修复说明模板（必须100%遵守）**：

```text
问题原因：
<用一句话说明问题产生的根本原因，精准定位到具体代码逻辑或数据条件>

修改描述：
<逐条列出修改内容，格式为"文件名: 修改动作"，每个文件占一行>

影响范围：
<列出受影响的功能模块和业务单据类型，多个用顿号分隔>

自测结果：
<描述本地验证的测试场景和对应结果，至少覆盖bug描述中的复现路径>

关联提交：
<可选，如果已提交代码，列出相关的commit hash或分支名，如"commit: abc1234"或"branch: fix/bug-xxx"；如果尚未提交则省略此项>

修复版本：
<可选，填写本次修复计划发布的版本号，如"v3.3.0"；如果尚未确定版本则省略此项>
```

**格式示例（正确）**：

```text
问题原因：
billing_goods_selector_submodel.dart中showsale参数仅在isPurchased为true时才传1，导致"全部商品"列表中showsale=0，后端不返回buyqty/returnqty字段

修改描述：
billing_goods_selector_submodel.dart: 将showsale参数逻辑从boolToValue(isPurchased)改为boolToValue(isPurchased || billUnion.customerId != null)

影响范围：
销售单开单、商品选择器

自测结果：
选择客户后切换到"全部商品"tab，商品sp00003正确显示买过/退过标签；无客户场景下标签不显示，行为保持不变

关联提交：
commit: a1b2c3d

修复版本：
v3.3.0
```

**格式反例（禁止）**：

```text
❌ 省略标题：直接写修复内容不带标题
❌ 修改标题名称：把"问题原因"改成"原因分析"或"Bug原因"
❌ 使用Markdown：用## 标题、- 列表、**加粗**等语法
❌ 添加额外标题：增加"备注"、"补充说明"等模板外的标题
❌ 合并标题：把多个标题合成一段
```

**评论提交流程**：

1. 按照上述模板生成修复说明草稿
2. **确认评论内容**：将生成的评论全文展示给用户审阅，使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交"
   - "修改内容"（用户通过"Other"选项提供修改意见，修改后再次确认）
3. 用户确认后，调用接口提交：

```text
mcp_RAM_create_work_item_comment 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
  - content: 用户确认后的纯文本评论内容（必填）
```

### 步骤六A：上传附件（可选）

当用户在步骤五选择"上传附件"时进入此流程：

1. 使用 `AskUserQuestion` 工具询问用户附件文件的本地绝对路径（用户通过“Other”选项输入文件路径）

2. 调用上传接口：
   ```text
   mcp_RAM_create_workitem_attachment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - filePath: 用户提供的本地文件绝对路径（必填）
   ```

3. 上传成功后，使用 `AskUserQuestion` 询问是否继续上传：
   - "继续上传"：回到本步骤第1步
   - "上传完成"：回到步骤五，继续选择后续操作（提交评论、更新状态等）

4. 如果上传失败，展示错误信息并询问是否重试

### 步骤七：更新工作项状态和属性

**查询工作流配置**：在更新状态前，先查询当前工作项类型允许的状态流转：

```text
mcp_RAM_get_work_item_workflow 参数说明：
  - organizationId: 组织ID（必填）
  - projectId: 项目ID，即步骤一获取的spaceId（必填）
  - workitemTypeId: 工作项类型ID，从步骤四获取的工作项详情中提取（必填）
```

**展示可选状态**：从工作流配置中解析当前状态允许流转到的目标状态列表，使用 `AskUserQuestion` 以选项形式展示合法的目标状态供用户选择。

如果工作流查询失败，降级为默认行为：直接建议“已修复(29)”作为目标状态。

**确认更新内容**：在调用API前，向用户展示即将更新的字段清单（如"状态: 处理中 → 已修复"），使用 `AskUserQuestion` 确认，选项包括：
- "确认更新"
- "取消"（取消则不更新状态，直接进入步骤八）

用户确认后，执行更新：

```text
mcp_RAM_update_work_item 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
  - updateWorkItemFields: 更新字段对象
      - status: 用户选择的目标状态ID
```

**重要说明**：工作项状态流转受云效工作流配置约束，可能无法通过API直接修改。如遇状态更新失败，提示用户在云效Web控制台手动操作。

### 步骤八：回到循环入口

当前Bug处理完毕（无论用户选择了提交评论、更新状态还是跳过），自动回到**步骤三**继续处理待处理列表中的下一个Bug。

无需额外询问"是否继续"，步骤三的"暂不处理"选项即为退出循环的唯一出口。

## 错误处理与重试机制

所有涉及云效API调用的步骤（评论提交、状态更新、负责人转交、附件上传等）均适用以下错误处理策略：

| 错误类型 | 典型表现 | 处理策略 |
|----------|----------|----------|
| 网络/临时错误 | 请求超时、服务端5xx错误 | 自动重试1次（间隔3秒），仍失败则提示用户稍后重试 |
| 权限/工作流错误 | 403禁止、状态流转被拒绝 | 不重试，展示具体限制原因，引导用户到云效Web控制台手动操作 |
| 参数错误 | 400请求无效、404资源不存在 | 不重试，展示错误详情，引导用户检查输入参数 |

**错误处理流程**：

1. API调用返回错误后，首先判断错误类型
2. 如果是网络/临时错误：
   - 提示用户"操作暂时失败，正在自动重试..."
   - 等待3秒后重试1次
   - 重试成功则继续后续流程
   - 重试仍失败则提示用户"操作失败，建议稍后重试或在云效Web端手动操作"
3. 如果是权限/工作流错误：
   - 提示用户具体的限制原因（如"当前状态不允许直接流转到已修复，请在云效Web端操作"）
   - 跳过当前操作，继续后续流程（不阻塞整体处理）
4. 如果是参数错误：
   - 展示错误详情
   - 询问用户是否要重新输入参数或跳过当前操作

**重要原则**：错误不应阻断整个Bug处理流程。即使某个API调用失败，也应允许用户继续处理其他步骤或其他Bug。

## 常见问题解答

**Q1：如何查询"我参与评论过的"工作项？**

A：云效API暂不支持此筛选条件，请登录云效Web控制台，在「工作项 → 我的」页面查看历史记录。

**Q2：工作项状态更新失败如何处理？**

A：状态流转受工作流严格限制，API可能无权限直接修改。建议先通过评论记录修复情况，然后在Web控制台手动更新状态。

**Q3：查询工作项时提示"项目不存在"怎么办？**

A：请确认 `spaceId` 参数正确。应先调用 `mcp_RAM_search_projects` 获取当前用户可访问的项目列表。
