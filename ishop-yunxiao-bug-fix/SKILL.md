# 云效DevOps工作项处理指南

## 核心原则

| 原则 | 说明 |
|------|------|
| 确认优先 | 不确定的事一定要问用户，不擅自做决定。所有涉及外部操作（修改代码、提交评论、更新状态、转交负责人）的步骤，必须先向用户展示具体内容并获得确认后才能执行 |
| 责任范围限定 | 仅处理分配给当前用户的Bug工作项，不主动介入他人工作项 |
| 评论格式约束 | 云效平台不支持Markdown渲染，评论内容必须使用纯文本格式 |
| 必填参数要求 | 查询工作项时必须提供 `spaceId`（项目ID）和 `category`（工作项类型）参数 |

## 操作流程

### 步骤一：获取基础信息

```text
mcp_RAM_get_current_organization_info  --> 获取组织ID(organizationId)和用户ID(userId)
mcp_RAM_search_projects                 --> 获取项目列表及对应的spaceId
```

**默认项目规则**：
- 如果用户**未指定具体项目名称**，默认使用 **"iShop ERP 项目"** 作为目标项目
- 调用 `mcp_RAM_search_projects` 时使用关键词 "iShop ERP" 搜索，获取对应的 `spaceId`
- 如果用户**明确指定了项目名称**，则按用户指定的项目查询
- 如果默认项目未找到，提示用户并询问要查看哪个项目

### 步骤二：查询工作项列表

查询当前用户分配的Bug工作项：

```text
mcp_RAM_search_workitems 参数说明：
  - organizationId: 组织ID（必填）
  - spaceId: 项目空间ID（必填）
  - category: "Bug"（必填，工作项类型）
  - assignedTo: "self" 或具体用户ID（指定负责人）
  - status: 状态ID列表，多个用逗号分隔（可选）
```

**默认状态过滤规则**：
- 默认只查询**需要处理的Bug**，即状态为：待确认(28)、待处理(100005)、处理中(100010)
- 查询时 `status` 参数默认传 `"28,100005,100010"`
- 已修复(29)和已关闭(100085)的Bug默认**不查询**，因为这些是已经处理过的
- 如果用户**明确要求**查看特定状态的Bug（如"看看已修复的bug"、"查看所有状态的bug"），则按用户指定的状态查询

**工作项状态ID对照表**：

| 状态名称 | 状态ID | 说明 | 默认查询 |
|----------|--------|------|----------|
| 待确认 | 28 | 新建Bug的初始状态 | ✅ |
| 待处理 | 100005 | 已确认待开发处理 | ✅ |
| 处理中 | 100010 | 开发人员正在修复 | ✅ |
| 已修复 | 29 | 修复完成待验证 | ❌ |
| 已关闭 | 100085 | 验证通过或无需修复 | ❌ |

### 步骤三：循环处理Bug列表（核心循环入口）

此步骤是Bug处理循环的入口，维护一个"待处理Bug列表"，每完成一个Bug后自动回到此步骤继续处理下一个，直到列表清空或用户主动退出。

**循环逻辑**：

1. **检查待处理列表**：
   - 列表为空 → 告知用户"所有Bug已处理完毕"，流程结束
   - 列表不为空 → 继续下方流程

2. **展示剩余Bug**：以简洁列表形式展示所有待处理Bug，每条包含：编号、标题、优先级、当前状态。如果是循环回来的（非首次），额外提示"已完成 N 个，剩余 M 个"

3. **询问用户选择**：使用 `AskUserQuestion` 工具，选项包括：
   - 各个Bug的标题作为选项（支持选择具体要修复的Bug）
   - "暂不处理"选项

4. **分支处理**：
   - 用户选择"暂不处理" → 流程结束
   - 用户选择了某个Bug → 将该Bug从待处理列表中移除，进入步骤四

### 步骤四：获取工作项详情并判断处理方式

```text
mcp_RAM_get_work_item 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项唯一标识（必填）
```

**获取附件（如果有的话）**：
```text
mcp_RAM_list_workitem_attachments 参数说明：
  - organizationId: 组织ID（必填）
  - workItemId: 工作项ID（必填）
```

获取详情后，向用户展示Bug的完整信息：

1. **基本信息**：标题、描述、复现步骤、优先级、创建人等
2. **附件展示**：
   - 如果有截图/图片附件，使用 `mcp_RAM_get_workitem_file` 获取下载链接
   - 展示图片给用户查看（使用 Markdown 图片语法：`![描述](图片URL)`）
   - 列出其他附件文件名和大小

**附件处理流程**：
1. 调用 `mcp_RAM_list_workitem_attachments` 获取附件列表
2. 遍历附件列表，识别图片类型（.png, .jpg, .jpeg, .gif, .webp 等）
3. 对于图片附件：
   - 调用 `mcp_RAM_get_workitem_file` 获取临时下载URL
   - 使用 Markdown 语法展示图片：`![Bug截图](下载URL)`
4. 对于其他附件：列出文件名、大小、上传时间

展示完详情后，使用 `AskUserQuestion` 工具询问处理方式，选项包括：

- **"修复此Bug"**：进入步骤四A（执行修复）
- **"不是我负责的，转给其他人"**：进入步骤四B（转交负责人）
- **"无法复现"**：进入步骤四C（退回创建人）
- **"不是问题/设计如此"**：进入步骤四D（标记为非问题）
- **"需要更多信息"**：进入步骤四E（退回创建人补充信息）

#### 步骤四A：执行修复

1. 分析Bug描述、复现步骤等信息
2. 在代码库中定位问题代码，形成修复方案
3. **确认修复方案**：向用户说明问题根因、修复思路、涉及的文件和改动点，使用 `AskUserQuestion` 工具确认是否按此方案执行，选项包括：
   - "确认，按此方案修复"
   - "我有其他想法"（用户通过"Other"选项补充修改意见）
4. 用户确认后，实施代码修复
5. 修复完成后，进入步骤五

#### 步骤四B：转交负责人（不是我负责的Bug）

当Bug内容不属于当前用户的职责范围时：

1. **询问是否需要添加评论**：使用 `AskUserQuestion` 工具询问"转交前是否需要添加评论说明原因？"，选项包括：
   - "需要添加评论"：进入步骤2的评论流程，评论完成后再继续转交流程
   - "自动生成评论并转交"：跳到步骤3，自动根据测试情况生成评论
   - "不添加评论，直接转交"：跳到步骤8继续（不推荐）

2. **评论流程（如果用户选择需要评论）**：
   - 使用 `AskUserQuestion` 工具询问用户评论内容（用户通过"Other"选项输入）
   - 展示生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 用户确认后的纯文本评论内容（必填）
     ```
   - 跳到步骤8继续转交流程

3. **自动生成评论（如果用户选择自动生成）**：
   - 根据Bug描述内容，自动生成转交说明评论
   - 评论内容应包含：
     - Bug涉及的功能模块/页面
     - 为什么不属于自己负责（如"此问题属于XX模块")
   - 展示自动生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交并转交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
     - "取消"（取消则进入步骤八，跳过此Bug）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 自动生成的转交说明评论内容（必填）
     ```

4. 使用 `AskUserQuestion` 工具询问用户："请输入要转交的负责人姓名"（用户通过"Other"选项输入自定义文本）

5. 根据用户提供的姓名，调用组织成员查询接口查找对应用户ID：
   ```text
   mcp_RAM_search_organization_members 参数说明：
     - organizationId: 组织ID（必填）
     - query: 用户提供的负责人姓名
   ```

6. 如果查到多个匹配结果，再次使用 `AskUserQuestion` 让用户从匹配列表中选择确切的人

7. **确认转交操作**：向用户展示即将执行的操作摘要（"将Bug [标题] 的负责人转交给 [新负责人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认转交"
   - "取消"（取消则进入步骤八，跳过此Bug）

8. 用户确认后，更新工作项负责人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 新负责人用户ID
   ```

9. 告知用户已成功转交，进入步骤八（继续处理下一个Bug）

**转交评论模板**：

```text
转交原因：
此问题属于[模块名称/功能名称]，由[负责人姓名/团队]负责处理。

问题描述：
<Bug标题或简要描述>

建议转交给：
[建议的负责人姓名]
```

#### 步骤四C：无法复现（退回创建人）

当Bug无法复现时，将Bug退回给创建人，并附上详细的复现尝试说明：

1. 从步骤四获取的工作项详情中提取**创建人ID**（creator字段）
2. 使用 `AskUserQuestion` 工具询问用户复现情况，提示用户描述：
   - 做过哪些测试操作
   - 测试的环境和条件
   - 为什么无法复现
   （用户通过"Other"选项输入自定义文本）
3. 根据用户描述，按照"无法复现说明模板"生成评论草稿
4. **确认评论内容和退回操作**：向用户展示生成的评论全文，并说明即将执行的操作（"提交以上评论，并将负责人转交给创建人 [创建人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并退回"
   - "修改内容"（用户通过"Other"选项提供修改意见，修改后再次确认）
   - "取消"（取消则进入步骤八，跳过此Bug）
5. 用户确认后，提交无法复现说明评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 按照下方"无法复现说明模板"填写的纯文本内容（必填）
   ```
6. 将负责人改为创建人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 创建人用户ID
   ```
7. 告知用户已退回给创建人，进入步骤八（继续处理下一个Bug）

**无法复现说明模板（必须100%遵守）**：

```text
复现结果：
无法复现

测试环境：
<描述测试使用的设备、系统版本、应用版本等环境信息>

测试操作：
<逐条列出按照Bug描述执行的测试步骤，格式为"N. 操作描述 -> 实际结果"，每步占一行>

结论：
<总结无法复现的原因判断，如"按照描述步骤多次测试均未出现该问题，建议提供更详细的复现条件"等>
```

**无法复现说明示例**：

```text
复现结果：
无法复现

测试环境：

测试操作：

2. 添加商品sp00003到购物车 -> 商品正确显示，价格和数量无异常
3. 点击提交订单 -> 订单提交成功，未出现描述中的闪退现象

结论：
按照Bug描述的复现步骤在Android 14设备上多次测试，均未出现闪退问题，建议提供具体的设备型号和操作系统版本以便进一步排查
```

#### 步骤四D：不是问题/设计如此

当Bug实际上是预期行为或设计如此时：

1. 使用 `AskUserQuestion` 工具询问用户说明原因（用户通过"Other"选项输入），提示用户描述：
   - 为什么这不是问题
   - 实际的业务逻辑是什么
   - 相关的设计文档或需求说明

2. 根据用户描述，按照"不是问题说明模板"生成评论草稿

3. **确认评论内容**：向用户展示生成的评论全文，使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并关闭"
   - "修改内容"（用户通过"Other"选项提供修改意见）
   - "取消"（取消则进入步骤八，跳过此Bug）

4. 用户确认后，提交评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

5. 告知用户已添加评论，进入步骤八（继续处理下一个Bug）

#### 步骤四E：需要更多信息

当Bug描述不够清晰，需要创建人补充信息时：

1. 从步骤四获取的工作项详情中提取**创建人ID**（creator字段）

2. 使用 `AskUserQuestion` 工具询问用户需要什么信息（用户通过"Other"选项输入），提示用户描述：

   - 需要创建人提供什么
   - 是否需要具体的复现条件

3. 根据用户描述，按照"需要更多信息模板"生成评论草稿

4. **确认评论内容和退回操作**：向用户展示生成的评论全文，并说明即将执行的操作（"提交以上评论，并将负责人转交给创建人 [创建人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并退回"
   - "修改内容"（用户通过"Other"选项提供修改意见）

5. 用户确认后，提交评论：
   ```text

     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

6. 将负责人改为创建人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 创建人用户ID
   ```

7. 告知用户已退回给创建人，进入步骤八（继续处理下一个Bug）

**需要更多信息模板**：

```text
状态：
需要更多信息

缺少的信息：
<逐条列出缺少的关键信息，如：
1. 具体的操作步骤不清晰，请补充完整流程
2. 缺少具体的错误截图或日志
3. 请提供复现该问题的具体条件
4. 请说明预期的正确行为是什么
>

请创建人补充以上信息后重新提交。
```

### 步骤五：修复完成后询问是否更新云效状态

代码修复完成后，使用 `AskUserQuestion` 工具询问用户后续操作，选项包括：
- "提交评论并更新状态为已修复"：执行步骤六（提交修复说明）+ 步骤七（更新状态为已修复，状态ID: 29），然后进入步骤八
- "仅提交评论"：仅执行步骤六（提交修复说明），不修改状态，然后进入步骤八
- "跳过"：不提交评论也不更新状态，直接进入步骤八（继续处理下一个Bug）

### 步骤六：提交修复说明（工作项评论）

**格式约束**：
- 评论内容必须为纯文本格式，不支持Markdown语法渲染
- 必须严格使用下方模板的四个固定标题，标题文字不可修改、不可省略、不可调换顺序
- 每个标题后紧跟冒号，冒号后换行书写具体内容
- 四个部分之间用一个空行分隔
- 禁止添加模板以外的标题或分节

**修复说明模板（必须100%遵守）**：

```text
问题原因：
<用一句话说明问题产生的根本原因，精准定位到具体代码逻辑或数据条件>

修改描述：
<逐条列出修改内容，格式为"文件名: 修改动作"，每个文件占一行>

影响范围：
<列出受影响的功能模块和业务单据类型，多个用顿号分隔>

自测结果：
<描述本地验证的测试场景和对应结果，至少覆盖bug描述中的复现路径>
```

**格式示例（正确）**：

```text
问题原因：
billing_goods_selector_submodel.dart中showsale参数仅在isPurchased为true时才传1，导致"全部商品"列表中showsale=0，后端不返回buyqty/returnqty字段

修改描述：
```

## 常见问题解答

**Q1：如何查询"我参与评论过的"工作项？**

A：云效API暂不支持此筛选条件，请登录云效Web控制台，在「工作项 → 我的」页面查看历史记录。

**Q2：工作项状态更新失败如何处理？**

A：状态流转受工作流严格限制，API可能无权限直接修改。建议先通过评论记录修复情况，然后在Web控制台手动更新状态。

**Q3：查询工作项时提示"项目不存在"怎么办？**

A：请确认 `spaceId` 参数正确。应先调用 `mcp_RAM_search_projects` 获取当前用户可访问的项目列表。
