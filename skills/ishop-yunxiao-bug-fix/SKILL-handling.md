# Bug处理方式详细流程

> 本文件是 ishop-yunxiao-bug-fix 技能的子文件（v2.3.0），包含步骤四A~四G的详细操作流程和评论模板。
> 由主文件 SKILL.md 在用户选择处理方式后通过 read_file 引用。

#### 步骤四A：执行修复

1. **更新状态为"处理中"**：如果当前Bug状态不是"处理中(100010)"，先将状态更新为"处理中"，表示开始修复：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - status: "100010"（处理中）
   ```
   如果状态更新失败（工作流限制），提示用户但不影响后续修复流程
2. 分析Bug描述、复现步骤等信息
3. 在代码库中定位问题代码，形成修复方案
4. **确认修复方案**：向用户说明问题根因、修复思路、涉及的文件和改动点，使用 `AskUserQuestion` 工具确认是否按此方案执行，选项包括：
   - "确认，按此方案修复"
   - "我有其他想法"（用户通过"Other"选项补充修改意见）
5. 用户确认后，实施代码修复
6. **修复确认循环**：代码修改完成后，使用 `AskUserQuestion` 工具向用户确认"代码修改是否已完成？"，选项包括：
   - "已完成，进入下一步"：退出循环，进入步骤五
   - "还没改完，继续修改"：用户通过 "Other" 选项描述还需要修改的内容，继续修改后再次确认，**重复此步骤直到用户确认完成**
7. 用户确认修复完成后，进入步骤五

#### 步骤四B：转交负责人（不是我负责的Bug）

当Bug内容不属于当前用户的职责范围时：

1. **询问是否需要添加评论**：使用 `AskUserQuestion` 工具询问"转交前是否需要添加评论说明原因？"，选项包括：
   - "需要添加评论"：进入步骤2的评论流程，评论完成后再继续转交流程
   - "自动生成评论并转交"：跳到步骤3，自动根据测试情况生成评论
   - "不添加评论，直接转交"：跳到步骤8继续（不推荐）

2. **评论流程（如果用户选择需要评论）**：
   - 使用 `AskUserQuestion` 工具询问用户评论内容（用户通过"Other"选项输入）
   - 展示生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 用户确认后的纯文本评论内容（必填）
     ```
   - 跳到步骤8继续转交流程

3. **自动生成评论（如果用户选择自动生成）**：
   - 根据Bug描述内容，自动生成转交说明评论
   - 评论内容应包含：
     - Bug涉及的功能模块/页面
     - 为什么不属于自己负责（如"此问题属于XX模块")
   - 展示自动生成的评论全文，使用 `AskUserQuestion` 确认：
     - "确认提交并转交"
     - "修改内容"（用户通过"Other"选项提供修改意见）
     - "取消"（取消则进入步骤八，跳过此Bug）
   - 用户确认后，提交评论：
     ```text
     mcp_RAM_create_work_item_comment 参数说明：
       - organizationId: 组织ID（必填）
       - workItemId: 工作项ID（必填）
       - content: 自动生成的转交说明评论内容（必填）
     ```

4. **智能推荐负责人**（自动执行）：
   - 分析当前Bug标题和描述中的模块/功能关键词（如"销售开单"、"商品列表"、"库存管理"等）
   - 调用 `mcp_RAM_search_workitems` 查询最近已修复（status=29）的包含相同关键词的Bug，提取其负责人信息
   - 如果找到推荐人选：使用 `AskUserQuestion` 展示推荐人选作为选项，同时提供"其他人"选项（用户通过"Other"输入自定义姓名）
   - 如果无法推荐（关键词不明确或无历史数据）：使用 `AskUserQuestion` 直接询问"请输入要转交的负责人姓名"（用户通过"Other"选项输入）

5. 根据用户提供的姓名，调用组织成员查询接口查找对应用户ID：
   ```text
   mcp_RAM_search_organization_members 参数说明：
     - organizationId: 组织ID（必填）
     - query: 用户提供的负责人姓名
   ```

6. 如果查到多个匹配结果，再次使用 `AskUserQuestion` 让用户从匹配列表中选择确切的人

7. **确认转交操作**：向用户展示即将执行的操作摘要（"将Bug [标题] 的负责人转交给 [新负责人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认转交"
   - "取消"（取消则进入步骤八，跳过此Bug）

8. 用户确认后，更新工作项负责人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 新负责人用户ID
   ```

9. 告知用户已成功转交，进入步骤八（继续处理下一个Bug）

**转交评论模板**：

```text
转交原因：
此问题属于[模块名称/功能名称]，由[负责人姓名/团队]负责处理。

问题描述：
<Bug标题或简要描述>

建议转交给：
[建议的负责人姓名]
```

#### 步骤四C：无法复现（退回创建人）

当Bug无法复现时，将Bug退回给创建人，并附上详细的复现尝试说明：

1. 从步骤四获取的工作项详情中提取**创建人ID**（creator字段）
2. 使用 `AskUserQuestion` 工具询问用户复现情况，提示用户描述：
   - 做过哪些测试操作
   - 测试的环境和条件
   - 为什么无法复现
   （用户通过"Other"选项输入自定义文本）
3. 根据用户描述，按照"无法复现说明模板"生成评论草稿
4. **确认评论内容和退回操作**：向用户展示生成的评论全文，并说明即将执行的操作（"提交以上评论，并将负责人转交给创建人 [创建人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并退回"
   - "修改内容"（用户通过"Other"选项提供修改意见，修改后再次确认）
   - "取消"（取消则进入步骤八，跳过此Bug）
5. 用户确认后，提交无法复现说明评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 按照下方"无法复现说明模板"填写的纯文本内容（必填）
   ```
6. 将负责人改为创建人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 创建人用户ID
   ```
7. 告知用户已退回给创建人，进入步骤八（继续处理下一个Bug）

**无法复现说明模板（必须100%遵守）**：

```text
复现结果：
无法复现

测试环境：
<描述测试使用的设备、系统版本、应用版本等环境信息>

测试操作：
<逐条列出按照Bug描述执行的测试步骤，格式为"N. 操作描述 -> 实际结果"，每步占一行>

结论：
<总结无法复现的原因判断，如"按照描述步骤多次测试均未出现该问题，建议提供更详细的复现条件"等>
```

**无法复现说明示例**：

```text
复现结果：
无法复现

测试环境：
Android 14 (Pixel 7), 应用版本 3.2.1, 测试账号 test_user_01

测试操作：
1. 进入销售开单页面，选择客户"测试客户A" -> 页面正常加载，无报错
2. 添加商品sp00003到购物车 -> 商品正确显示，价格和数量无异常
3. 点击提交订单 -> 订单提交成功，未出现描述中的闪退现象
4. 重复以上步骤5次 -> 均未复现

结论：
按照Bug描述的复现步骤在Android 14设备上多次测试，均未出现闪退问题，建议提供具体的设备型号和操作系统版本以便进一步排查
```

#### 步骤四D：不是问题/设计如此

当Bug实际上是预期行为或设计如此时：

1. 使用 `AskUserQuestion` 工具询问用户说明原因（用户通过"Other"选项输入），提示用户描述：
   - 为什么这不是问题
   - 实际的业务逻辑是什么
   - 相关的设计文档或需求说明

2. 根据用户描述，按照"不是问题说明模板"生成评论草稿

3. **确认评论内容**：向用户展示生成的评论全文，使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并关闭"
   - "修改内容"（用户通过"Other"选项提供修改意见）
   - "取消"（取消则进入步骤八，跳过此Bug）

4. 用户确认后，提交评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

5. 告知用户已添加评论，进入步骤八（继续处理下一个Bug）

**不是问题说明模板**：

```text
分析结果：
不是问题/设计如此

问题描述：
<复述用户反馈的问题现象>

实际行为：
<说明实际的业务逻辑和预期行为>

原因说明：
<说明为什么这是预期行为，如"这是按设计文档XXX的要求实现的"或"这是业务规则XXX的要求"等>

建议：
如有疑问，请联系产品经理确认需求
```

#### 步骤四E：需要更多信息

当Bug描述不够清晰，需要创建人补充信息时：

1. 从步骤四获取的工作项详情中提取**创建人ID**（creator字段）

2. 使用 `AskUserQuestion` 工具询问用户需要什么信息（用户通过"Other"选项输入），提示用户描述：
   - 缺少哪些关键信息
   - 需要创建人提供什么
   - 是否需要具体的复现条件

3. 根据用户描述，按照"需要更多信息模板"生成评论草稿

4. **确认评论内容和退回操作**：向用户展示生成的评论全文，并说明即将执行的操作（"提交以上评论，并将负责人转交给创建人 [创建人姓名]"），使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交并退回"
   - "修改内容"（用户通过"Other"选项提供修改意见）
   - "取消"（取消则进入步骤八，跳过此Bug）

5. 用户确认后，提交评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

6. 将负责人改为创建人：
   ```text
   mcp_RAM_update_work_item 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - updateWorkItemFields:
         - assignedTo: 创建人用户ID
   ```

7. 告知用户已退回给创建人，进入步骤八（继续处理下一个Bug）

**需要更多信息模板**：

```text
状态：
需要更多信息

缺少的信息：
<逐条列出缺少的关键信息，如：
1. 具体的操作步骤不清晰，请补充完整流程
2. 缺少具体的错误截图或日志
3. 请提供复现该问题的具体条件
4. 请说明预期的正确行为是什么
>

请创建人补充以上信息后重新提交。
```

#### 步骤四F：重复Bug

当Bug与已有的其他Bug重复时：

1. 使用 `AskUserQuestion` 工具询问用户"请提供原始Bug的编号或标题"（用户通过"Other"选项输入）

2. 根据用户提供的信息，按照"重复Bug说明模板"生成评论草稿

3. **确认评论内容**：向用户展示生成的评论全文，使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交"
   - "修改内容"（用户通过"Other"选项提供修改意见）
   - "取消"（取消则进入步骤八，跳过此Bug）

4. 用户确认后，提交评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

5. 告知用户已标记为重复Bug，进入步骤八（继续处理下一个Bug）

**重复Bug说明模板**：

```text
分析结果：
重复Bug

原始Bug：
<原始Bug的编号和标题>

重复说明：
<简要说明为什么认为是重复的，如"与原始Bug描述的问题现象一致，均为XX功能的XX问题">

建议：
请关闭此Bug，跟踪原始Bug的修复进度
```

#### 步骤四G：延期处理

当Bug确实存在但当前版本无法修复，需要延期到后续版本时：

1. 使用 `AskUserQuestion` 工具询问用户延期原因（用户通过"Other"选项输入），提示用户描述：
   - 延期的原因（如"当前版本封版"、"需要整体重构"、"依赖其他功能开发"等）
   - 预计处理的版本或时间

2. 根据用户描述，按照"延期处理说明模板"生成评论草稿

3. **确认评论内容**：向用户展示生成的评论全文，使用 `AskUserQuestion` 确认，选项包括：
   - "确认提交"
   - "修改内容"（用户通过"Other"选项提供修改意见）
   - "取消"（取消则进入步骤八，跳过此Bug）

4. 用户确认后，提交评论：
   ```text
   mcp_RAM_create_work_item_comment 参数说明：
     - organizationId: 组织ID（必填）
     - workItemId: 工作项ID（必填）
     - content: 用户确认后的纯文本评论内容（必填）
   ```

5. 告知用户已标记为延期处理，进入步骤八（继续处理下一个Bug）

**延期处理说明模板**：

```text
处理结果：
延期处理

延期原因：
<说明为什么需要延期，如"当前版本已封版，修复该问题需要较大范围的重构">

预计处理时间：
<说明预计在哪个版本或时间节点处理，如"v3.3.0版本"或"下个迭代周期">

当前影响：
<说明延期对用户的影响程度和临时解决方案（如有）>
```
